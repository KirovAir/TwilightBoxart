Browser support and what this page does to your card.

Which browsers can write to your card

Browser Pick a folder Write to the card Notes
Chrome, Edge, Opera, Brave, Vivaldi (desktop) Yes Yes, in place The intended experience
Firefox (all versions) No No Mozilla standards position #154: negative, closed. No flag exists.
Safari, desktop No No WebKit standards position #28: oppose
Anything on iOS / iPadOS No No Every iOS browser is WebKit underneath
Chrome on Android Partial Not usefully Android hands back content URIs with no atomic writes

Writing into a folder you chose needs the File System Access API, which only Chromium browsers implement. Mozilla and WebKit have both formally declined it; these are published positions, not backlog items, so do not wait for this to change.

The fallback everywhere else

On a browser without the API this page still works, read-only:

1. Pick your card folder with the ordinary file chooser. The page reads it; it cannot write to it.
2. It identifies your ROMs and downloads the art exactly as it would otherwise.
3. You get one .zip. Extract it at the root of the card; the folders inside are already _nds/TWiLightMenu/boxart/, so everything lands in the right place.

The only thing you lose is the last step being automatic. The desktop app has no such limitation.

Install it as an app

If the browser offers Install app, take it. An installed app keeps permission to your card folder between launches, so you pick the folder once, ever. An ordinary tab loses the grant when its last window closes, and Chrome stops offering the "remember this" prompt after three dismissals.

Installation caches only the page itself, never anything from the API. A cached identity or a stale rendered PNG would end up written to your card and look permanent, which is not a failure mode worth the offline convenience.

What it does to your card

It writes PNG files into _nds/TWiLightMenu/boxart/ (or the folder you set manually), creating it if it is missing, and it reads a few hundred bytes from each game to work out what it is. It changes nothing else. No renaming, no moving, no deleting, and specifically no rewriting of settings.ini: the server guarantees every image fits TWiLightMenu's cache slot, so nothing needs patching.

Every write goes through a swap file that is only swapped in on close, so pulling the card mid-run cannot leave a half-written image behind. Images that already exist are left alone unless you tick "Overwrite existing boxart?".

Why some ROMs get nothing

Click through the misses list; each row says why. The common reasons:

Not in the index. Romhacks, translations and homebrew often have nothing to match against; roughly 60% of No-Intro entries carry no serial.
No art exists upstream. The ROM was identified, but GameTDB and libretro-thumbnails have no cover for it.
A .7z with a compressed header. Browsers have no LZMA decoder. Re-pack those as .zip, or use the desktop client.
NES, SNES, Master System and Game Gear loose files. These have no internal serial, so the scan checksums them automatically (files up to 64 MB) and matches by content. A miss here usually means a romhack or a dump the index does not know.
